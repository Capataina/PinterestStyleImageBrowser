# INT8-quantised CLIP/SigLIP encoder variant + per-EP benchmark matrix

## What the addition is

Two coupled additions:

1. **A static-INT8-quantised version of the chosen image encoder**. Use HF Optimum to produce `models/model_image_int8.onnx` from the FP32 model, with calibration on a 200-500-image sample of the project's corpus. Wire as a config flag (`[encoder] precision = "fp32" | "int8"`).
2. **A per-EP benchmark matrix** at `bench-results/ep_compat.md` documenting which combinations of encoder × precision × ONNX EP work, with measured per-image latency. Concretely: rows = {OpenAI CLIP-B/32, SigLIP-2-Base, MobileCLIP-S2}, columns = {CPU FP32, CPU INT8, CUDA FP32, CoreML FP16}, cells = {latency_ms, status: ok / error / quality-degraded}.

## Audience targeted

**Primary: A1 Rust + applied-ML systems engineers** — `audience.md` Audience 1 signal-function: "Hardware portability: CUDA + CoreML + CPU all proven to load; per-EP fallback table; documented kernel gaps". The matrix IS that fallback table. The user's own commits already document the CoreML transformer-op gap (`90a3842`, `2775c9f`) — turning that diagnosis into a published table is the credibility marker.

**Secondary: A4** — INT8-vs-FP32 cosine preservation is part of a complete embedding-quality story.

## Coupling-grade classification

**Plug-and-play** — the INT8 model is a separate ONNX file. The encoder impl reads the precision config and loads the matching file. Default stays FP32 unless the user opts in. The bench matrix is a documentation artefact + a `cargo bench` target; no runtime cost.

## Integration plan

**The project today is a local-first Tauri 2 desktop app for browsing and semantically searching local image libraries with CLIP via ONNX Runtime, organised around a `CosineIndex` brute-force similarity engine over SQLite-stored f32 BLOBs, with documented CoreML EP gaps for transformer ops.** This rec adds an INT8 model variant + a published per-EP / per-precision matrix.

```
   Today: one model per encoder        After Rec-6: precision-tier
   ───────────────────────────         ──────────────────────────
   models/model_image.onnx              models/model_image_fp32.onnx (rename)
                                        models/model_image_int8.onnx (new)
                                        models/model_image_fp16.onnx (optional)

   Encoder construction: load whichever variant matches config.precision.
```

The `bench-results/ep_compat.md` is generated by a `xtask bench-eps` runner that iterates encoder × precision × EP, records latency + a small cosine-vs-FP32-CPU sanity number, writes the table. Run periodically (manually); CI optional.

## Anti-thesis

This recommendation would NOT improve the project if:

- The user runs only on Apple Silicon and CoreML is the dominant target — INT8 on CoreML has its own gaps, mileage varies.
- The corpus is too small for meaningful calibration (<200 images). Then dynamic quantisation is the fallback.
- The project ships only on hardware where Encoder latency is a non-issue (e.g., always-CUDA workstation). Then INT8 is solving a non-problem.
- The chosen encoder (per Rec-3) is already MobileCLIP-class small; further quantisation has diminishing returns.

## Implementation cost

**Small: 3-5 days.**

Milestones:
1. Set up the Optimum quantisation script (Python). ~½ day.
2. Curate a calibration sample (200-500 images, representative). ~½ day.
3. Run quantisation; verify cosine-vs-FP32 ≥ 0.99 on a held-out set. ~1 day.
4. Add config plumbing to load INT8 model when `[encoder] precision = "int8"`. ~½ day.
5. Implement the `xtask bench-eps` runner — small Rust binary that iterates the EP × precision matrix and writes a markdown table. ~1 day.
6. Generate the matrix on the user's hardware (Mac Silicon, since that's the user's primary platform per profile). ~½ day.
7. Document the matrix + the CoreML transformer-op gap in `context/notes/onnx-ep-compatibility.md`. ~½ day.

Required reading before starting: `context/notes/clip-preprocessing-decisions.md` — the FP32 baseline must already pass the cosine ≥ 0.999 vs Python reference test before INT8 quantisation is meaningful.
